# 索引结构

知识地图的索引负责导航，不负责承载长篇事实。

## `docs/index.md`

公共入口必须包含：

- 知识地图用途：说明它是人和智能体共用的工作环境。
- 先读顺序：链接到 `docs/map/index.md`、`docs/map/schema.md`、`docs/map/workflows.md`。
- 目录分区表：列出每个顶层目录和职责。
- 事实优先级：说明任务路由、领域归属、实现落点、约束和运行手册分别以哪些节点为准。
- 如果仓库存在 `docs/wiki/`，目录分区表必须说明它是代码浓缩解释层，并提供 wiki 入口或首批关键目录链接。

## `docs/map/index.md`

控制入口必须包含：

- 阅读顺序：通常从仓库指导文件到 schema、不变量、工作流，再到任务相关节点。
- 工作环境分层表：控制层、格式层、领域层、工作流层、约束层、记忆层、质量层、参考层。
- 如果启用 wiki，工作环境分层表应包含 Wiki 层，并说明它服务于代码目录检索和链路理解。
- 常用入口：知识查询、代码定位、系统理解、代码编写/审查、知识地图维护。

## `docs/map/workflows.md`

工作流索引必须包含：

- 任务类型到工作流文件的映射表。
- 每条工作流的 `KM:*` 链接。
- “先选工作流，再读最小必要节点”的使用规则。
- 必须包含 `docs/workflows/impact-map.md`，用于开工前输出 repository impact map。

## `docs/map/domains.md`

领域索引必须包含：

- 领域到领域节点的映射表。
- 每个领域的职责摘要。
- 领域落到具体文件时应读取的参考索引。

## `docs/references/indexes/`

实现索引负责实现落点，不承担长篇解释。索引条目应尽量是：

```text
路径 + 归属 + 用途 + 验证依据
```

## `docs/wiki/index.md`

wiki 入口必须包含：

- wiki 的用途：说明它是贴近代码目录的浓缩解释层。
- 目录映射：列出已维护的 wiki 目录和对应代码路径。
- 阅读规则：先读父级 wiki，再按链路下钻到关键子目录、reference/index 和代码。
- 维护规则：代码目录变动时同步 wiki，稳定概念再沉淀到 domain/tag/glossary。
